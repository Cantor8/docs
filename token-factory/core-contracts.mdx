Token Factory contracts fall into two groups: **balance contracts** (holdings) and **operational contracts** (factory, instructions, allocations, config).

Represents funds locked for an in-flight transfer or allocation.

</ParamField>

</ParamField>

<ParamField path="fields" type="admin, owner, amount, lockHolder, expiresAt, createdAt">

`lockHolder` is the **receiver** for transfer offers, or the **executor** for allocations.

## AutoAccept

Opt-in flag that lets a receiver accept direct transfers in a single transaction.

<ParamField path="signatories" type="admin, receiver">
</ParamField>

<ParamField path="choice" type="AutoAccept_Cancel (controlled by receiver)">
</ParamField>

If the receiver has an `AutoAccept` contract, a transfer creates the receiver holding directly. Otherwise the transfer creates a two-step offer.

### Factory choices

| Choice | Controller | Purpose |
| --- | --- | --- |
| `Mint` | admin | Create a new holding for a recipient. |
| `Burn` | admin | Burn holdings, usually for bridge withdrawal or redemption. |
| `AdminBurn` | admin | Force-burn holdings for compliance or emergency workflows. Requires a reason. |
| `Migrate` | admin | Archive an old holding and create a current-version holding. |
| `CreateAutoAccept` | receiver | Create the receiver's auto-accept flag. |
| `MergeSplit` | owner | Split or merge holdings while preserving precision rules. |
| `AdminArchive` | admin | Archive the factory itself. |

### CIP-0056 interfaces

Factories also implement two CIP-0056 interfaces:

- **`TransferFactory`** — prepare and initiate transfers.
- **`AllocationFactory`** — prepare allocations for atomic settlement and DvP workflows.

## TransferInstruction

Created when a receiver does **not** have `AutoAccept`.

<ParamField path="signatories" type="admin, validator, sender">
</ParamField>

<ParamField path="observer" type="receiver">
</ParamField>

<ParamField path="backed by" type="LockedHolding">
</ParamField>

| Choice | Controller | Purpose |
| --- | --- | --- |
| `TransferInstruction_Accept` | receiver | Accept transfer, archive the lock, create receiver holding. |
| `TransferInstruction_Reject` | receiver | Reject transfer and return funds to sender. |
| `TransferInstruction_Withdraw` | sender | Withdraw after expiry and return funds to sender. |
| `AdminForceBurn` | admin \+ validator | Burn locked funds through a dual-control proposal/approval flow. |

## TokenAllocation

Locks funds for atomic settlement.

<ParamField path="signatories" type="admin, validator, sender">
</ParamField>

<ParamField path="observers" type="executor, receiver">
</ParamField>

<ParamField path="backed by" type="LockedHolding">
</ParamField>

| Choice | Controller | Purpose |
| --- | --- | --- |
| `Allocation_ExecuteTransfer` | executor \+ sender \+ receiver | Execute settlement and transfer funds to receiver. |
| `Allocation_Withdraw` | sender | Withdraw after expiry. |
| `Allocation_Cancel` | executor \+ sender \+ receiver | Cancel and return funds. |
| `AdminForceBurn` | admin \+ validator | Burn locked allocation funds through the dual-control compliance flow. |
