## Admin authority

Holdings use **`admin` as sole signatory** and **`owner` as observer**. Token spending is intentionally routed through factory choices rather than owner-local choices.

## Dual-control force burn

Locked funds in transfer instructions and allocations use an **`AdminForceBurn`** flow controlled by **both `admin` and `validator`**.

The implementation uses a **proposal / approval pattern** so the exercise carries both authorities. A non-empty reason is required and is emitted under the standard splice reason metadata key.

<Info>
  Dual control means no single party can destroy locked funds. Both admin and validator must sign.
</Info>

## Freeze enforcement

Freeze is enforced **at contract execution time**, not only when a workflow is created.

<CardGroup cols={2}>
  <Card title="Blocked by freeze" icon="ban">
    - Mint
    - Burn
    - Merge / split
    - Transfer initiation
    - Transfer accept
    - Allocation create
    - Allocation execute
  </Card>

  <Card title="Allowed during freeze" icon="circle-check">
    - Reject
    - Withdraw
    - Cancel
    - AdminBurn
    - AdminForceBurn
  </Card>
</CardGroup>

<Tip>
  Parties can always return funds and admin teams can always complete compliance actions — even while a token is frozen.
</Tip>

## Decimal precision

Decimal precision is defined **in each token's holding package** and enforced whenever new holdings are created.

| Token | Decimals | Smallest amount |
| --- | --: | --- |
| `c8ETH` | 10 | `0.0000000001` |
| `c8BTC` | 8 | `0.00000001` |
| `c8TEST` | 10 | `0.0000000001` |
| `rCC` | 10 | `0.0000000001` |

Precision is validated for **mint, transfer, allocation, merge / split outputs, and change amounts**.
